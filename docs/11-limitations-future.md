# 11. Limitations และ Future Improvement Plan

> **หลักของเอกสารนี้:** ทุกข้อจำกัดต้องมาคู่กับ **สิ่งที่เราทำแล้วเพื่อไม่ให้มันหลอกผู้ใช้** และ **สิ่งที่จะแก้จริงถ้าไปต่อ** — ไม่ใช่รายการข้อแก้ตัว
>
> ข้อจำกัดหลายข้อในนี้ **ไม่ใช่การเดา** แต่เจอจริงระหว่างสร้าง (มีเลข error, เลข commit, ผลทดสอบกำกับ)

## 11.1 ขอบเขตที่ตั้งใจไม่ทำ

| ไม่ทำ | เหตุผล |
| --- | --- |
| นับแคลอรี · วิเคราะห์รูปอาหาร · อะไรที่แตะน้ำหนัก/รูปร่าง | **ห้ามโดย guardrail ไม่ใช่แค่ตัดออกเพราะไม่มีเวลา** — `findForbiddenTerms()` บล็อกข้อความที่ระบบสร้างเอง และ CI แดงถ้าหลุด |
| Push notification / LINE | ต้องมี infra ส่งข้อความ + ขอ consent เพิ่ม เกินสโคป 4 สัปดาห์ |
| เชื่อม wearable / health API | ไม่มีอุปกรณ์ให้ทดสอบ และจะเปลี่ยนดีไซน์ check-in ทั้งหมด |
| Dashboard ภาพรวมองค์กร/มหาวิทยาลัย | ขัดหลัก privacy ที่ประกาศไว้ ถ้าทำต้องเป็นค่าเฉลี่ยกลุ่ม ≥ 20 คนเท่านั้น |
| รองรับหลาย persona | โจทย์ข้อ 7.1 ให้เลือก persona เดียว — เลือก "ปาล์ม" (นักศึกษาปี 3 ตารางไม่แน่นอน) |

## 11.2 ข้อจำกัดของข้อมูล

**ข้อมูลมาจากผู้ใช้กรอกเองทั้งหมด (self-reported)** — ไม่มีเซนเซอร์ยืนยัน คนจำผิดหรือกรอกให้ดูดีได้
→ **ทำแล้ว:** ออกแบบให้การกรอกตามจริงไม่มีต้นทุนทางใจ — ไม่มีคะแนน ไม่มีเกรด ไม่มีสตรีคที่ทำให้เสียดายเวลาขาด ภาษาไม่ตัดสิน
→ **ถ้าไปต่อ:** ดึงชั่วโมงนอน/จำนวนก้าวจาก wearable มาเป็นค่าตั้งต้นให้ผู้ใช้แก้ ลดทั้งภาระและอคติ

**เก็บ "นาทีเคลื่อนไหวรวม + ชนิดกิจกรรม" แต่ไม่เก็บนาทีต่อชนิด**
→ เจอตอนทำกราฟ F2-02: โค้ดเวอร์ชันแรกเอานาทีรวมหารเฉลี่ยทุกชนิดแล้ววาดเป็นสัดส่วน (เดิน 25 + ยืด 5 นาที กลายเป็น 15/15) = **กราฟโกหก** → เปลี่ยนเป็นแท่งนาทีรวมสีเดียวที่ตรงกับข้อมูลจริง
→ **ทำไมไม่เก็บเพิ่ม:** ถามนาทีต่อชนิดคือเพิ่มคำถามในฟอร์มที่ต้องจบใน 1–3 นาที **แลกไม่คุ้ม** — เราเลือกกราฟที่จริงกว่าแทนกราฟที่ละเอียดกว่า

**ข้อมูลยังสั้น** — demo account 24 วัน + ทีม dogfood ตั้งแต่ 13 ก.ค.
→ **ทำแล้ว:** `checkDataSufficiency()` ปฏิเสธการวิเคราะห์ถ้าน้อยกว่า 7 วัน (F3-04) และ reflection ต่ำกว่า 3 วันจะไม่เรียก AI เลย — **ระบบยอมบอกว่า "ยังไม่พอ" ดีกว่าเดาให้ฟังดูฉลาด**

**ยังไม่มีการประเมินผลกับผู้ใช้จริงนอกทีม** — n = 4 และทุกคนคือคนสร้าง
→ **ถ้าไปต่อ:** ทดสอบกับนักศึกษา 10–15 คนที่ไม่รู้จักโปรเจกต์ วัด 2 อย่าง: กรอกจบใน 3 นาทีจริงไหม และสัปดาห์ที่ 2 ยังกลับมาบันทึกไหม

## 11.3 ข้อจำกัดของ AI

**Pattern ที่เจอคือความสัมพันธ์ ไม่ใช่เหตุและผล**
→ **ทำแล้ว 3 ชั้น:** โค้ดไม่ส่งข้อมูล < 7 วันให้ LLM · system prompt สั่งห้ามสรุปเหตุ-ผลพร้อมตัวอย่าง · `CAUSAL_TERMS` (`เป็นเพราะ`, `เกิดจาก`, `สาเหตุมาจาก`) เป็นคำที่ตรวจจับได้
→ **หลักฐาน:** QA-01 เคส 10 รอบแรก AI หลุดเขียน "อาจจะเป็นเพราะตื่นสาย" → แก้ prompt → รอบยืนยัน 16 ก.ค. นำเสนอเป็น "ข้อสังเกต" แล้ว

**คำแนะนำมาจาก LLM ที่ควบคุมได้ไม่ 100%**
→ **ทำแล้ว:** ทุก goal ต้องผ่าน `validateGoalTitle()` ก่อนถูกใช้เสมอ ไม่ผ่าน = ทิ้งแล้ว fallback ไป goal มาตรฐานจากตารางโจทย์ · **เป็นการรับประกันเชิงโครงสร้าง ไม่ใช่การสุ่มตรวจ**
→ **ข้อจำกัดที่เหลือ:** guardrail ของ**ข้อความแชท**ยังต้องใช้คนอ่านยืนยัน เพราะการปฏิเสธที่ถูกต้องต้องเอ่ยคำต้องห้ามเอง ("โค้ชให้แผนลดน้ำหนักไม่ได้") — บล็อกอัตโนมัติ = โค้ชปฏิเสธไม่เป็น

**บทสนทนาใน guided flow เป็นบทที่เขียนไว้ล่วงหน้า ไม่ใช่ AI**
→ ตัว flow (ถาม 3 คำถาม) เป็นแบบฟอร์มที่แต่งเป็นบทสนทนา · **ส่วนที่เป็น AI จริงคือ goal ที่เสนอ** ซึ่งมาจาก check-in จริง + โปรไฟล์ + คำตอบใน flow ผ่าน `recommendGoals(context)`
→ **ถ้าไปต่อ:** ให้ LLM คุมบทสนทนาเองแบบ multi-turn — แต่ต้องแลกกับความคาดเดาได้และโควตา

**ผลทดสอบ safety ผ่านคือ "รอบที่สะอาด" ไม่ใช่การพิสูจน์ทางคณิตศาสตร์** — LLM เป็นระบบความน่าจะเป็น
→ **ทำแล้ว (QA-01 ปิดสมบูรณ์ 19 ก.ค.):** รันเต็ม 10 เคส × 2 ประโยค = 20/20 นัด บนโมเดล production ที่ pin ไว้ · ประโยคทดสอบ commit ก่อนรัน · ผลดิบ commit ไม่ตัดต่อ · **คนตรวจยืนยันไม่ใช่คนเขียน prompt** (หลักฐานใน `.scratch/ai-safety-test/`)
→ **ข้อจำกัดที่เหลือ:** ผ่าน 20/20 หนึ่งรอบไม่การันตีทุกรอบ — จึงมีด่านโค้ด (`validateGoalTitle`, escalation, sufficiency) เป็นชั้นที่ไม่ขึ้นกับความน่าจะเป็นของ LLM

## 11.4 ข้อจำกัดทางเทคนิคที่กระทบผู้ใช้จริง

**Gemini free tier มีเพดานโควตาทั้งแอปรวมกัน** (ADR-0003 + INFRA-23)
→ **เจอชนเพดานจริง 2 ครั้งใน 24 ชม.** ระหว่างพัฒนา (16–17 ก.ค. ตอนใช้ `gemini-2.5-flash` = 20 ครั้ง/วัน) — ไม่ใช่ความเสี่ยงบนกระดาษ
→ **ทำแล้ว:** ย้ายโมเดลเป็น `gemini-3.1-flash-lite` = **500 ครั้ง/วัน** (25 เท่า — INFRA-23 · รัน safety checklist ใหม่ครบบนโมเดลใหม่) · cache-first ทุกผลวิเคราะห์ · จำกัดแชท 5 ข้อความ/คน/วัน · **fallback ปลอดภัยทุกจุดเมื่อ AI ล่ม** (ผู้ใช้ยังได้ goal/สรุปจากตารางมาตรฐาน ไม่เห็นหน้าพัง) · อุ่นผลวิเคราะห์ demo ล่วงหน้าด้วยคำสั่งเดียว
→ **ข้อจำกัดที่เหลือ:** 500/วันพอสำหรับ demo + ทีม แต่ไม่พอถ้ามีผู้ใช้จริงหลักสิบ (แชร์โควตาก้อนเดียว) + เพดานนาที 15 req/min
→ **ถ้าไปต่อ:** ขึ้น paid tier (ระดับ $ ไม่กี่ดอลลาร์/เดือนก็พอสำหรับผู้ใช้หลักร้อย) + แยกโควตารายผู้ใช้

**ล้างประวัติแชท = โควตา 5 ข้อความ/วันรีเซ็ต** (`countMessagesToday()` นับแถวใน DB ที่ลบได้)
→ เจอจริงตอนเพื่อนกดล้างประวัติระหว่างทดสอบแล้วโควตาเด้งกลับ · **ตามอยู่ใน INFRA-20** — ถ้าไม่ทันก่อน freeze จะยังอยู่ในเวอร์ชันที่ demo

**สรุปสัปดาห์ที่ cache ไว้อาจเก่ากว่าบันทึกล่าสุด** — ข้อความ AI ถูกสร้างตอนหนึ่ง แล้วผู้ใช้บันทึกเพิ่มทีหลัง
→ **ทำแล้ว:** ไม่กลบ — หน้า reflection ขึ้นการ์ด "สรุปนี้ยังไม่รวมบันทึกล่าสุดของคุณ" พร้อมปุ่มสร้างใหม่ · ตัวเลขเทียบสัปดาห์คำนวณสดเสมอ
→ **ถ้าไปต่อ:** regenerate อัตโนมัติแบบ background job — ตอนนี้ทำไม่ได้เพราะจะกินโควตาโดยผู้ใช้ไม่ได้ขอ

**เป็น responsive web app ไม่ใช่แอปเนทีฟ** (ADR-0001) — ไม่มี widget, ไม่มี background sync, ไม่มี local notification
→ ยอมรับเพื่อให้ 4 คนส่งของได้ทันใน 4 สัปดาห์ และแชร์ลิงก์ให้กรรมการเปิดได้ทันที

**คำตอบโค้ชมี markdown ดิบโผล่บ้าง** (`**ข้อความ**`) — ติดตามใน F4-06

## 11.5 สิ่งที่เรียนรู้จากการสร้างและใช้จริง

> ส่วนนี้ไม่มีอยู่ในแผนตอนแรก — เกิดจากการที่ทีม dogfood และรีวิวโค้ดกันเองจริงจัง

**1. "มีเทสต์" ไม่เท่ากับ "จับได้"** — มี PR ที่เปลี่ยนโมเดล AI จาก `gemini-2.5-flash` เป็น `gemini-2.0-flash` แล้ว **CI ผ่านครบทั้ง 4 ด่าน** ถ้า merge ไป หลักฐานความปลอดภัยทั้งหมดจะกลายเป็นของโมเดลที่ไม่เคยทดสอบ โดยไม่มีใครรู้จนวัน pitch
→ **แก้:** unit test ล็อกชื่อโมเดล (INFRA-19) พร้อมข้อความบอกว่าหลักฐานอะไรจะพังถ้าเปลี่ยน — **ตอนนี้การเปลี่ยนโมเดลเงียบ ๆ เป็นไปไม่ได้แล้ว**

**2. เทสต์จับหน้าจอที่พัง แต่ไม่จับหน้าที่ "ไปไม่ถึง"** — มี PR ที่ถอดการ์ดออกจาก dashboard ซึ่งบังเอิญเป็น**ลิงก์เดียวในทั้งแอปที่ไปหน้าสรุปสัปดาห์** e2e ยังเขียวหมดเพราะมันเปิดหน้านั้นผ่าน URL ตรง ๆ
→ **แก้:** จับได้ตอนรีวิวด้วยคน + คืนการ์ดกลับ · บทเรียนคือ **หน้าที่ไม่มีทางเข้า = หน้าที่ไม่มีอยู่จริง** สำหรับผู้ใช้และกรรมการ

**3. บัญชี demo ที่เป็นสนามทดสอบด้วย จะมีขยะโผล่ตอนนำเสนอ** — เจอข้อความทดสอบ (`ddsdsa`) และ goal ทดสอบค้างในบัญชีปาล์ม
→ **แก้:** ออกกติกา "เทสด้วยบัญชีสมัครทิ้ง" + ใส่การล้างบัญชี demo ในเช็คลิสต์ก่อน pitch (QA-03)

**4. เอกสารที่ตามหลังโค้ดคือเอกสารที่หลอกทีมตัวเอง** — มีช่วงที่บอร์ดอ่านว่างานยังไม่เริ่ม ทั้งที่โค้ดขึ้น production ไปแล้ว (เพราะ PR ใบเดียวทำ 2 issue) และมีการอ้างถึง issue ที่ไม่เคยถูกสร้าง
→ **แก้:** ปิด issue พร้อมหลักฐานทุกใบ · sync บอร์ดทุกครั้งที่ merge เป็นชุด

**5. ฟอร์มแคบกรอกง่ายกว่า แต่เดสก์ท็อปจะดูโล่ง** — แก้ด้วยการเติม step rail ข้าง ๆ **ไม่ใช่ขยายฟอร์ม** (INFRA-17)
