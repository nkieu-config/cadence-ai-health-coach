# 04 — Requirements (SRS ฉบับย่อ)

## Functional Requirements

รหัส FR ตาม feature ของโจทย์ ระดับความสำคัญ: **M** = Must (ต้องมีถึง demo ได้), **S** = Should (ควรมี), **C** = Could (ถ้าทัน)

### F0 — Account & Onboarding

| รหัส | Requirement | ระดับ |
|---|---|---|
| FR-0.1 | สมัคร/ล็อกอินด้วย **Google OAuth** และ email+password (Supabase Auth) — ดู ADR-0005 | M |
| FR-0.2 | Onboarding เก็บบริบทชีวิต (โจทย์ 5.1): สถานะ (นักศึกษา/first jobber), วันที่มีเรียน/งานเช้า, **ช่วงที่งานหนัก/มี deadline** (`busy_periods`), ข้อจำกัด (รวม "เดินทางไป-กลับนาน") | M |
| FR-0.3 | แสดง safety disclaimer ("wellness coach ไม่ใช่แพทย์") ตอน onboarding และผู้ใช้ต้องกดรับทราบ | M |

### F1 — Low-Burden Daily Check-in

| รหัส | Requirement | ระดับ |
|---|---|---|
| FR-1.1 | บันทึก check-in ได้วันละ 1 รายการ ครอบคลุมโจทย์ 5.2–5.4: มื้ออาหาร (จำนวน + มื้อที่ข้าม + **เวลามื้อแรก** + **ของว่าง/ผัก-ผลไม้** + เครื่องดื่มหวาน), ชั่วโมงนอน + เวลาเข้านอน + คุณภาพ + เหตุผลที่นอนดึก, energy level, การเคลื่อนไหว (ประเภท + นาที + อุปสรรค), disruptor, note สั้น | M |
| FR-1.2 | ทุกคำถามเป็นตัวเลือก/ปุ่ม กรอกจบใน ≤ 3 นาที — วัดจริงตอน dogfooding (QA-02) · คำถามเสริมโผล่เฉพาะเมื่อเกี่ยว (เช่น ถามเวลามื้อแรกเฉพาะวันที่ได้กิน) | M |
| FR-1.3 | แก้ไข check-in ย้อนหลังได้ และบันทึกย้อนหลังได้ 1 วัน | S |
| FR-1.4 | ความรู้สึกหลังมื้ออาหาร (`meal_feeling`) **และหลังเคลื่อนไหว** (`movement_feeling`) เป็นตัวเลือกเสริม ข้ามได้ — โจทย์ 5.2 + 5.4 | S |
| FR-1.5 | **เวลาตื่นนอน (โจทย์ 5.3) ไม่ถามซ้ำ** — คำนวณจาก `bed_time_bucket` + `sleep_hours` แล้วแสดงผล (`wakeTimeRange()`) เพื่อรักษา Low Burden | S |

### F2 — Lifestyle Dashboard

| รหัส | Requirement | ระดับ |
|---|---|---|
| FR-2.1 | แสดงสรุป 7/14/30 วันของ 3 pillars: กราฟชั่วโมงนอน, มื้อที่กิน/ข้าม, นาทีเคลื่อนไหว, energy level | M |
| FR-2.2 | ทุกกราฟ overlay วันที่มี disruptor ให้เห็นความเชื่อมโยงกับตารางชีวิต | M |
| FR-2.3 | แสดงตาราง pattern → ความหมาย → next step ตามรูปแบบในโจทย์ Feature 2 | M |
| FR-2.4 | แสดง streak การบันทึก (เชิงบวก ไม่ลงโทษวันที่ขาด) | C |

### F3 — Pattern Analysis

| รหัส | Requirement | ระดับ |
|---|---|---|
| FR-3.1 | ระบบคำนวณ pattern candidates จากข้อมูลจริงด้วยโค้ด (ไม่ใช่ให้ LLM เดาเลขเอง) เช่น ค่าเฉลี่ย energy แยกตามกลุ่มวันนอน &lt;6 ชม. vs ≥6 ชม. | M |
| FR-3.2 | ส่ง pattern candidates + บริบทให้ Gemini แปลเป็นภาษา insight ที่ไม่ตัดสิน และแยก "สัญญาณ" ออกจาก "ข้อสรุป" เสมอ | M |
| FR-3.3 | ถ้าข้อมูล &lt; 7 วัน แสดงข้อความว่ายังไม่พอวิเคราะห์ พร้อมชวนบันทึกต่อ — ห้ามมโน pattern | M |
| FR-3.4 | Cache ผลวิเคราะห์ใน DB, refresh เมื่อมี check-in ใหม่ | S |

### F4 — AI Coach Conversation

| รหัส | Requirement | ระดับ |
|---|---|---|
| FR-4.1 | หน้าแชทคุยกับ coach ได้ โดย coach มี context: ข้อมูล check-in ล่าสุด, pattern, micro goal ปัจจุบัน | M |
| FR-4.2 | ทุกคำตอบผ่าน system prompt guardrail กลาง (docs/08) | M |
| FR-4.3 | มี guided flow อย่างน้อย 1 flow: "ตั้ง micro goal สัปดาห์หน้า" — coach ถามนำ 2–3 คำถามแล้วสรุปเป็น goal | M |
| FR-4.4 | ถ้าผู้ใช้พูดถึงอาการผิดปกติ/ความกังวลสุขภาพจิต coach ต้องแนะนำให้ปรึกษาผู้เชี่ยวชาญ ไม่วินิจฉัยต่อ | M |
| FR-4.5 | ประวัติแชทเก็บใน DB และผู้ใช้ลบได้ | S |

### F5 — Micro Goal Recommendation

| รหัส | Requirement | ระดับ |
|---|---|---|
| FR-5.1 | ระบบเสนอ micro goal 1–2 ข้อ/สัปดาห์ อิงสถานการณ์จริงจาก check-in + disruptor ตามตารางในโจทย์ Feature 5 | M |
| FR-5.2 | ผู้ใช้กดรับ/ปรับ/ปฏิเสธ goal ได้ และติ๊กความคืบหน้ารายวันได้ | M |
| FR-5.3 | Goal ห้ามเกี่ยวกับน้ำหนัก รูปร่าง แคลอรี หรือการอด — บังคับทั้งใน prompt และ validation | M |

### F6 — Weekly Reflection

| รหัส | Requirement | ระดับ |
|---|---|---|
| FR-6.1 | สร้างรายงานรายสัปดาห์: ความถี่บันทึก, สรุป 3 pillars เชื่อมกับ disruptor, จุดแข็งที่ควรรักษา, ข้อเสนอ 1–2 ข้อสัปดาห์หน้า ตามรูปแบบโจทย์ Feature 6 | M |
| FR-6.2 | ดูรายงานย้อนหลังได้ | S |
| FR-6.3 | ภาษาเชิงบวก ไม่ให้คะแนนสุขภาพ ไม่ตำหนิ | M |

### F7 — Privacy Controls

| รหัส | Requirement | ระดับ |
|---|---|---|
| FR-7.1 | ผู้ใช้เห็นเฉพาะข้อมูลตัวเอง (Supabase RLS ทุกตาราง) | M |
| FR-7.2 | หน้า privacy: อธิบายว่าเก็บอะไร ใช้อย่างไร + ปุ่มลบข้อมูลทั้งหมด/ลบบัญชี | M |
| FR-7.3 | แก้ไข/ลบ check-in รายรายการได้ | M |

## Non-Functional Requirements

| รหัส | Requirement |
|---|---|
| NFR-1 | Responsive ใช้ได้ทั้งมือถือและ desktop (demo ทั้งสองแบบตอน pitch) |
| NFR-2 | AI ตอบภายใน ~10 วินาที มี loading state; ถ้า Gemini ล่ม/ติด rate limit แสดงข้อความสุภาพ ไม่ crash |
| NFR-3 | ภาษา UI และ AI ทั้งหมดเป็นภาษาไทย |
| NFR-4 | API key อยู่ฝั่ง server เท่านั้น ไม่มี secret ใน client bundle |
| NFR-5 | ต้นทุนรัน = 0 บาท (free tier ทั้งหมด) |
| NFR-6 | เก็บข้อมูลแบบ data minimization — ไม่เก็บน้ำหนัก ส่วนสูง รูปถ่าย หรือข้อมูลที่ไม่ได้ใช้ |

## เกณฑ์ตัดสินเมื่อเวลาไม่พอ

ตัดตามลำดับ: C ก่อน → S → ถ้า M ไม่ทันให้ลด scope ภายใน M (เช่น coach เหลือ guided flow เดียว) — **ห้ามตัด FR-0.3, FR-3.3, FR-4.4, FR-5.3, FR-7.x** เพราะเป็นเกณฑ์ให้คะแนน Safety/Privacy โดยตรง
