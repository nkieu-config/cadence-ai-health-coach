# 08 — Safety Guardrail & Privacy Design

ตอบ Mission ข้อ 8 (Responsible AI and Safety) และข้อ 9 (Privacy) — สองหัวข้อนี้เป็นเกณฑ์ให้คะแนนโดยตรงและ**ห้ามตัดออกเมื่อเวลาไม่พอ** (ดู docs/04 ท้ายไฟล์)

## Part 1 — Safety Guardrail

### ขอบเขตของระบบ

ระบบคือ **wellness coach** ช่วยสังเกต pattern และเสนอก้าวเล็ก ๆ — ไม่ใช่เครื่องมือแพทย์ ไม่วินิจฉัย ไม่รักษา

### ชั้นการป้องกัน 3 ชั้น

| ชั้น | กลไก | ครอบคลุม |
|---|---|---|
| 1. UI | Disclaimer บังคับกดรับทราบตอน onboarding + ข้อความกำกับถาวรแทบทุกหน้า (3 layout รวม landing/login): "Cadence เป็นผู้ช่วยดูแลสุขภาพทั่วไป ไม่ใช่บริการทางการแพทย์ — หากมีอาการผิดปกติควรปรึกษาผู้เชี่ยวชาญ" | ผู้ใช้เข้าใจขอบเขตก่อนใช้ |
| 2. System prompt | ทุก AI call ผ่าน `lib/ai` (เนื้อหาเต็มใน [07-ai-design.md](07-ai-design.md)) · coach = prompt เต็ม (การแพทย์/escalation/ไม่ตัดสิน) · insight/goal/reflection = prompt เฉพาะทาง (ห้ามน้ำหนัก/รูปร่าง/แต่งเลข/ตัดสิน) | AI output ทุกชิ้น |
| 3. Code validation | `findForbiddenTerms()` กรอง insight/goal/reflection (goal: reject → retry 2 รอบ → fallback ที่เขียนไว้) · ข้อมูล < 7 วัน → ไม่เรียก pattern analysis เลย | insight/goal/reflection (ดู note แชทใต้ตาราง) |

> **คำตอบแชทของโค้ช** ผ่านชั้น 2 (coach prompt) แต่จงใจไม่ผ่านชั้น 3 (code filter) — เพราะการปฏิเสธที่ถูกต้องมักต้องเอ่ยคำต้องห้ามเอง (เช่น "ไม่แนะนำการอดอาหาร") การกรองด้วยคำจะ false-positive ใส่คำตอบที่ดี · ความปลอดภัยของแชทยืนยันด้วย **QA-01: 20/20 บนโมเดล production** ([หลักฐาน](../.scratch/ai-safety-test/))

### กติกาเนื้อหา (บังคับทุก AI output)

- ไม่วินิจฉัยโรค ไม่แนะนำยา อาหารเสริม หรือการรักษา
- ไม่ให้แผนลดน้ำหนัก ไม่พูดถึงรูปร่าง น้ำหนัก หน้าตา
- ไม่สนับสนุนการอดอาหารหรือออกกำลังกายหักโหม
- แยก "pattern จากข้อมูล" กับ "ข้อสรุป" — ใช้ภาษา "สัญญาณที่น่าติดตาม" เสมอ
- ภาษาไม่ตัดสิน ไม่ทำให้รู้สึกผิด แม้ผู้ใช้ขาดบันทึกหรือทำ goal ไม่สำเร็จ

### Escalation

เมื่อผู้ใช้พูดถึงอาการผิดปกติทางกาย ความเครียดรุนแรง หรือการทำร้ายตัวเอง → coach แสดงความห่วงใย แนะนำพบแพทย์/นักจิตวิทยา และให้เบอร์**สายด่วนสุขภาพจิต 1323** — ไม่วิเคราะห์อาการต่อ (FR-4.4, ทดสอบใน checklist docs/07)

## Part 2 — Privacy Design

### ระบบเก็บอะไร (ตอบโจทย์ข้อ 9 ตรง ๆ)

| ข้อมูล | จัดเป็น | เก็บเพื่อ |
|---|---|---|
| email | ข้อมูลส่วนบุคคล | login เท่านั้น |
| display_name, สถานะ, วันเรียนเช้า, ข้อจำกัด | ข้อมูลส่วนบุคคล/บริบท | personalize คำแนะนำ |
| check-in (กิน นอน เคลื่อนไหว พลังงาน disruptor) | **ข้อมูลสุขภาพ** | pattern analysis, dashboard, reflection |
| ประวัติแชทกับ coach | **ข้อมูลสุขภาพ** | ความต่อเนื่องของบทสนทนา |

สิ่งที่**ไม่เก็บ**: น้ำหนัก ส่วนสูง BMI แคลอรี รูปถ่าย ตำแหน่ง — data minimization ตั้งแต่ design (docs/05)

### สิทธิของผู้ใช้

- **ดู** — เห็นข้อมูลตัวเองทั้งหมดใน dashboard/history
- **แก้ไข** — แก้ check-in ย้อนหลังได้ทุกรายการ
- **ลบ** — ลบ check-in รายรายการ (หน้า history) · ลบประวัติแชท (หน้า coach) · ลบข้อมูลทั้งหมดหรือลบบัญชีถาวร (หน้า settings/privacy) — ได้เองทุกอย่างไม่ต้องร้องขอ · ลบบัญชีเป็น cascade delete ทุกตาราง

### การป้องกันทางเทคนิค

- **Supabase RLS ทุกตาราง**: policy `for all to authenticated using (auth.uid() = user_id)` (migration 0003 — ปิด role `anon` ด้วย) — ผู้ใช้เข้าถึงได้เฉพาะแถวของตัวเอง แม้ API มีบั๊กก็ข้ามข้อมูลคนอื่นไม่ได้
- Gemini ได้รับเฉพาะข้อมูลพฤติกรรม **ไม่แนบชื่อ/email** และระบบไม่ใช้ข้อมูลบันทึกในโปรเจกต์นี้เพื่อเทรนโมเดลเพิ่มเติม
- Secrets ทั้งหมดฝั่ง server (NFR-4), การเชื่อมต่อทั้งหมดผ่าน HTTPS

### หลักการเผื่ออนาคต (บันทึกใน limitations)

ถ้ามี dashboard ภาพรวมสำหรับมหาวิทยาลัย/องค์กร ต้องเป็น aggregate เท่านั้น (เช่น ค่าเฉลี่ยกลุ่ม ≥ 20 คน) ห้าม drill down ถึงรายบุคคล — อยู่นอก scope prototype นี้

### ข้อความหน้า Privacy ในแอป

> Cadence เก็บบันทึกการกิน การนอน และการเคลื่อนไหวที่คุณกรอกเอง เพื่อแสดง pattern และให้คำแนะนำสำหรับคุณเท่านั้น ข้อมูลของคุณไม่ถูกแชร์ให้ผู้ใช้คนอื่นหรือบุคคลที่สาม คุณแก้ไขหรือลบข้อมูลทั้งหมดได้ทุกเมื่อจากหน้านี้ ระบบนี้เป็นผู้ช่วยดูแลสุขภาพทั่วไป ไม่ใช่บริการทางการแพทย์
